# TGV2RayScraper

TGV2RayScraper - это проект на Python, предназначенный для сбора данных из Telegram-каналов, извлечения и обработки конфигураций V2Ray, включая их очистку, нормализацию и удаление дубликатов. Проект поддерживает актуальную информацию о каналах и включает инструменты для управления их списками. Он предоставляет асинхронные инструменты для сбора данных и синхронные инструменты для обработки конфигураций V2Ray.

> Проект запускается на Python версии 3.10 и выше.

Для английской версии смотрите [README.md](../../README.md)

## Быстрый старт

### Клонирование репозитория

Склонируем проект на свой компьютер:

```bash
git clone https://github.com/denxv/TGV2RayScraper.git
```

Переходим в директорию проекта:

```bash
cd TGV2RayScraper
```

### Работа с командой `uv`

> Все команды `uv` одинаково работают на Linux, macOS и Windows.

#### Создание виртуального окружения

Создаём и активируем виртуальное окружение:

```bash

```

## Использование

### **1. Обновление каналов**

Вы можете запустить скрипт обновления каналов следующим образом:

```bash
python -m scripts.update_channels
```

> Вы также можете добавить `uv run` перед любой командой `python`, чтобы запустить её через `uv`.

Также доступен альтернативный способ запуска с использованием `PYTHONPATH`:

```bash
PYTHONPATH=. python scripts/update_channels.py
```

Можно использовать `-h`, чтобы увидеть все доступные опции:

```bash
python -m scripts.update_channels -h
```

**Опции**

* **Глобальные опции**

  * `--debug` - Включить отладочное логирование в консоли. По умолчанию в консоли отображаются логи уровня `INFO`.

  * `--no-dry-run` - Отключить режим проверки и разрешить изменение метаданных каналов, включая присвоение `current_id` и сброс полей (например, `count`, `last_id` и т.д.). По умолчанию включён режим проверки (`dry-run`).

  * `--skip-backup` - Пропустить создание резервных копий файлов каналов и Telegram URL перед сохранением. По умолчанию резервные копии создаются.

* **Входные файлы**

  * `-C, --channels PATH` - Путь к JSON-файлу со списком каналов (по умолчанию: `channels/current.json`).

  * `-U, --urls PATH` - Путь к входному TXT-файлу с новыми URL каналов (по умолчанию: `channels/urls.txt`).

* **Выбор каналов**

  * `-F, --channel-filter CONDITION` - Фильтр каналов в виде Python-подобного выражения (например: `"count < 100 and current_id == last_id or state == -1"`).

    > Используется для выбора каналов, к которым применяются операции сброса (`--reset-*`).

* **Действия с каналами**

  * `-D, --delete-channels` - Удалить каналы, которые недоступны или соответствуют определённым условиям. По умолчанию удаление отключено.

  * `-M, --message-offset N` - Количество последних сообщений, учитываемых при назначении `current_id`.

* **Сброс значений каналов**

  * `--reset-all` - Сбросить все поля каналов к значениям по умолчанию (можно комбинировать с `--reset-<field>` и `--channel-filter`).

  * `--reset-count [N]` - Сбросить `count` в указанное значение или в значение по умолчанию (`0`).

  * `--reset-current-id [N]` - Сбросить `current_id` в указанное значение или в значение по умолчанию (`1`).

  * `--reset-last-id [N]` - Сбросить `last_id` в указанное значение или в значение по умолчанию (`-1`).

  * `--reset-state [N]` - Сбросить `state` в указанное значение или в значение по умолчанию (`0`).

**Скрипт выполняет следующие действия:**

* Отображает в консоли логи уровня `INFO` по умолчанию, отладочный вывод включается через параметр `--debug`.

* Логирует подробную отладочную информацию и предупреждения в файл `logs/yyyy-mm-dd.log`.

* Загружает текущий список каналов из файла `channels/current.json`.

* Объединяет существующие каналы с новыми URL из файла `channels/urls.txt`.

* По умолчанию работает в режиме пробного запуска без внесения изменений (режим можно отключить с помощью `--no-dry-run`).

* При необходимости создаёт резервные копии файлов со списком каналов и URL (резервное копирование можно пропустить с помощью `--skip-backup`).

* Отбирает каналы на основе заданного фильтра (`--channel-filter`), если он указан.

* Удаляет недоступные или помеченные каналы при использовании опции `--delete-channels`.

* Сбрасывает значения полей каналов к значениям по умолчанию или к указанным значениям (`--reset-*`).

* Назначает или обновляет поле `current_id` с учётом указанного смещения сообщений (`--message-offset`).

* Сохраняет обновлённые данные обратно в файлы `channels/current.json` и `channels/urls.txt`.

**Пример использования:**

```bash
python -m scripts.update_channels -C channels/current.json -U channels/urls.txt -F "count < 100" --no-dry-run --reset-all
```

> Можете добавить `uv run` перед командой `python`, чтобы запустить её через `uv`.

### **2. Запуск скраперов**

#### **Асинхронный скрапер** (быстрый и стабильный)

Вы можете запустить асинхронный скрапер следующим образом:

```bash
python -m scripts.scraper
```

> Вы также можете добавить `uv run` перед любой командой `python`, чтобы запустить её через `uv`.

Также доступен альтернативный способ запуска с использованием `PYTHONPATH`:

```bash
PYTHONPATH=. python scripts/scraper.py
```

Можно использовать `-h`, чтобы увидеть все доступные опции:

```bash
python -m scripts.scraper -h
```

**Опции**

* **Глобальные опции**

  * `--debug` - Включить отладочное логирование в консоли. По умолчанию в консоли отображаются логи уровня `INFO`.

* **HTTP-клиент**

  * `--proxy [URL]` - Прокси-сервер для HTTP-запросов. Имеет приоритет над переменными окружения. Если не указан, используются `HTTPS_PROXY`, `HTTP_PROXY`, `ALL_PROXY`. Если ничего не найдено, используется локальный прокси по умолчанию (`socks5://127.0.0.1:10808`).

    * Поддерживаемые протоколы: `http`, `https`, `socks5`, `socks5h`.

    * Формат: `protocol://[username:password@]host:port`.

  * `--retries N` - Максимальное количество повторных попыток HTTP-запроса при ошибках (по умолчанию: `3`).

  * `--retry-delay SECONDS` - Максимальное количество повторных попыток HTTP-запроса после неудачных попыток (по умолчанию: `0.5`).

  * `--time-out SECONDS` - Таймаут HTTP-клиента в секундах для запросов, используемых при обновлении информации о каналах и извлечении V2Ray-конфигураций (по умолчанию: `30.0`).

* **Входные и выходные файлы**

  * `-C, --channels PATH` - Путь к входному JSON-файлу со списком каналов (по умолчанию: `channels/current.json`).

  * `-R, --configs-raw PATH` - Путь к выходному TXT-файлу для сохранения собранных V2Ray-конфигураций (по умолчанию: `configs/v2ray-raw.txt`).

* **Обновление каналов**

  * `--skip-update` - Пропустить обновление информации о каналах. Позволяет избежать лишних запросов, если каналы уже обновлены. По умолчанию обновление каналов выполняется.

  * `-U, --channels-batch N` - Количество каналов, обрабатываемых за один пакет при обновлении (по умолчанию: `100`).

* **Извлечение конфигураций**

  * `-E, --configs-batch N` - Количество сообщений, обрабатываемых за один пакет при извлечении конфигураций (по умолчанию: `20`).

  * `-P, --channels-concurrency N` - Максимальное количество каналов, одновременно обрабатываемых при извлечении конфигураций (по умолчанию: `5`).

**Скрипт выполняет следующие действия:**

* Отображает в консоли логи уровня `INFO` по умолчанию, отладочный вывод включается через параметр `--debug`.

* Логирует подробную информацию о процессе обновления и извлечения, включая предупреждения и ошибки для каждого канала в файл `logs/yyyy-mm-dd.log`.

* Загружает текущий список каналов из файла `channels/current.json`.

* Параллельно обновляет метаданные каналов (если не использован `--skip-update`), число одновременно обновляемых задаётся через `--channels-batch`.

* Параллельно извлекает V2Ray-конфигурации, используя:

  * `--channels-concurrency` - максимальное количество каналов, которые обрабатываются одновременно;

  * `--configs-batch` - количество сообщений, обрабатываемых за один шаг внутри одного канала.

* Выполняет все сетевые запросы через прокси-сервер, указанный в параметре `--proxy`.

* Использует HTTP-клиент с таймаутом `--time-out` для всех запросов, включая обновление каналов и извлечение конфигураций.

* При ошибках сетевых запросов использует повторные попытки (`--retries`) с задержкой между ними (`--retry-delay`).

* Сохраняет извлечённые V2Ray-конфигурации в файл `configs/v2ray-raw.txt`.

**Пример использования:**

```bash
python -m scripts.scraper -C channels/current.json -R configs/v2ray-raw.txt -E 20 -U 100 --proxy socks5://127.0.0.1:10808 --time-out 30.0 --skip-update
```

> Можете добавить `uv run` перед командой `python`, чтобы запустить её через `uv`.

### **3. Очистка конфигураций V2Ray**

Вы можете запустить скрипт очистки конфигураций V2Ray следующим образом:

```bash
python -m scripts.v2ray_cleaner
```

> Вы также можете добавить `uv run` перед любой командой `python`, чтобы запустить её через `uv`.

Также доступен альтернативный способ запуска с использованием `PYTHONPATH`:

```bash
PYTHONPATH=. python scripts/v2ray_cleaner.py
```

Можно использовать `-h`, чтобы увидеть все доступные опции:

```bash
python -m scripts.v2ray_cleaner -h
```

**Опции**

* **Глобальные опции**

  * `--debug` - Включить отладочное логирование в консоли. По умолчанию в консоли отображаются логи уровня `INFO`.

  * `--skip-normalize` - Пропустить нормализацию конфигураций, чтобы сохранить их исходную структуру. По умолчанию нормализация включена.

* **Входные файлы**

  * `-I, --configs-raw PATH` - Путь к входному TXT-файлу с необработанными V2Ray-конфигурациями для парсинга (по умолчанию: `configs/v2ray-raw.txt`).

  * `--import [PATH]` - Путь к входному JSON-файлу с уже распарсенными конфигами. Если файл пустой или недействительный, будут распарсены необработанные конфиги (по умолчанию: `configs/v2ray.json`).

* **Выходные файлы**

  * `-O, --configs-clean PATH` - Путь к выходному TXT-файлу для очищенных и обработанных конфигов (по умолчанию: `configs/v2ray-clean.txt`).

  * `--export [PATH]` - Путь к выходному JSON-файлу для экспорта распарсенных конфигов для последующего использования без повторного парсинга (по умолчанию: `configs/v2ray.json`).

* **Фильтрация / Сортировка**

  * `-D, --duplicate [FIELDS]` - Удалять дубликаты по указанным полям (по умолчанию: `"protocol, host, port"`).

  * `-F, --config-filter CONDITION` - Оставлять только записи, подходящие под Python-подобное условие (например: `"host == '1.1.1.1' and port > 1000"`).

  * `-R, --reverse` - Сортировать в порядке убывания (только с `--sort`).

  * `-S, --sort [FIELDS]` - Сортировка по полям через запятую (по умолчанию: `"protocol"`).

**Скрипт выполняет следующее:**

* Отображает в консоли логи уровня `INFO` по умолчанию, отладочный вывод включается через параметр `--debug`.

* Логирует подробную информацию о процессе обработки, включая ошибки и предупреждения для каждой конфигурации в файл `logs/yyyy-mm-dd.log`.

* Читает сырые V2Ray-конфигурации из файла `configs/v2ray-raw.txt` и выполняет их парсинг для последующей обработки.

* Импортирует уже распарсенные конфигурации из JSON-файла через опцию `--import`. Если указанный файл пустой или недействительный, используется парсинг сырых конфигов.

* Применяет фильтры на основе Python-подобных условий с помощью параметра `--config-filter` и выполняет опциональную нормализацию, которую можно пропустить через `--skip-normalize`.

* Удаляет дубликаты по указанным полям при использовании опции `--duplicate`.

* Сортирует записи по указанным полям с помощью `--sort` и при необходимости меняет порядок на обратный через `--reverse`.

* Сохраняет очищенные и обработанные конфигурации в файл `configs/v2ray-clean.txt`.

* Экспортирует распарсенные конфигурации в JSON-файл через опцию `--export` для последующего повторного использования без повторного парсинга.

* Поддерживает гибкий выбор полей для фильтрации, сортировки и удаления дубликатов, что позволяет извлекать только нужные конфигурации.

**Пример использования:**

```bash
python -m scripts.v2ray_cleaner -I configs/v2ray-raw.txt -O configs/v2ray-clean.txt -F "re_search(r'speedtest|google', host)" --reverse -D "host, port" -S "protocol, host, port" --import configs/v2ray.json --export
```

> Можете добавить `uv run` перед командой `python`, чтобы запустить её через `uv`.

### **4. Запуск всех операций через `main.py`**

```bash
python main.py
```

> Вы также можете добавить `uv run` перед любой командой `python`, чтобы запустить её через `uv`.

Можно использовать `-h`, чтобы увидеть все доступные опции:

```bash
python main.py -h
```

**Опции**

* **Глобальные опции**

  * `-D, --debug` - Включить отладочное логирование в консоли. По умолчанию в консоли отображаются логи уровня `INFO`.

  * `-H, --help-scripts [NAMES]` - Показать справку по внутренним скриптам pipeline. Можно указать имена скриптов через запятую (например: `"scraper, update_channels, v2ray_cleaner"`). Если значение не указано (например: `-H`), справка выводится для всех скриптов.

**Скрипт выполняет следующее:**

* Выполняет все этапы pipeline в порядке:

  1. `update_channels.py` – обновляет список каналов.

  2. `scraper.py` – собирает данные каналов из Telegram асинхронно.

  3. `v2ray_cleaner.py` – очищает, нормализует и обрабатывает полученные прокси-конфигурации.

* Автоматически передаёт каждому скрипту только релевантные аргументы.

**Пример использования:**

```bash
python main.py --debug --configs-batch 10 --channels-batch 50 --channels-concurrency 10 --proxy --retry-delay 1.5 --channel-filter "state == -1" --config-filter "host and port" --duplicate --reset-all --reset-count 0 --reverse --sort "host, port"
```

## Зависимости

### Основные зависимости

Проект требует следующие библиотеки Python (работает с Python 3.10+):

* **aiofiles** – асинхронная работа с файлами

* **asteval** – безопасная оценка Python-выражений (используется для фильтрации каналов и конфигураций)

* **httpx[socks]** – современный HTTP-клиент с поддержкой асинхронных запросов и SOCKS-прокси (включает зависимость `socksio`)

* **lxml** – парсинг и обработка HTML/XML

* **rich** – форматированный вывод в терминал, цветной логгинг и улучшенное отображение прогресса

Полный список зависимостей доступен в [`requirements.txt`](../../requirements.txt).

### Dev-зависимости (для разработки)

Для разработки и тестирования проекта требуются дополнительные инструменты:

* **mypy** – проверка типов

* **pytest** – фреймворк для тестирования

* **pytest-asyncio** – поддержка асинхронных тестов в `pytest`

* **pytest-cov** – отчёты о покрытии тестами

* **pytest-mock** – мокирование в тестах

* **ruff** – статический анализ и линтинг кода

Все dev-зависимости перечислены в [`requirements-dev.txt`](../../requirements-dev.txt).

## Структура проекта

* **adapters/** - адаптеры для асинхронной работы с внешними данными и Telegram

  * `channel.py` - асинхронные HTTP-запросы к веб-превью Telegram, парсинг HTML через `lxml`, извлечение ID постов, загрузка/сохранение JSON/URL и создание бэкапов

  * `config.py` - асинхронное извлечение сообщений, парсинг V2Ray-ссылок регулярными выражениями, управление прогресс-барами, импорт/экспорт конфигов в TXT/JSON

  * `scraper.py` - оркестратор обновления метаданных каналов: батчинг, конкурентная обработка, интеграция с рендерерами `rich`

* **channels/** - рабочее хранилище состояния пула каналов

  * `current.json` - основной JSON-файл с метаданными каналов (`count`, `current_id`, `last_id`, `state`)

  * `urls.txt` - исходный список ссылок на Telegram-каналы для добавления в пул

  * *также хранятся автоматические бэкапы файлов (например, `current-backup-<timestamp>.json`)*

* **configs/** - директория для собранных и обработанных конфигураций

  * `v2ray-clean.txt` - итоговый файл с очищенными, нормализованными и отфильтрованными конфигурациями

  * `v2ray-raw.txt` - сырые конфигурации, напрямую извлечённые скрейпером из постов

  * `v2ray.json` - JSON-кэш распарсенных конфигураций для ускорения повторной обработки

* **core/** - ядро проекта: утилиты, константы, типы и инфраструктура

  * **constants/** - статические данные, паттерны, шаблоны строк и конфигурационные лимиты

    * **messages/** - текстовые константы для логирования, разбитые по уровням

      * `error.py` - сообщения об ошибках

      * `info.py` - информационные сообщения

      * `warning.py` - предупреждения

    * **patterns/** - скомпилированные регулярные выражения

      * **v2ray/** - паттерны для парсинга URL-форматов протоколов

        * `common.py` - общие паттерны V2Ray (например, JSON-детектор)

        * `detector.py` - универсальный паттерн для определения протокола и тела ссылки

        * `registry.py` - реестр паттернов, сгруппированных по протоколам

        * `url.py` - специфичные паттерны для разбора полей каждого протокола

      * `common.py` - паттерны полей конфигов и разделителей параметров

      * `proxy.py` - валидация URL прокси-серверов

      * `telegram.py` - извлечение имён каналов из ссылок `t.me`

    * **templates/** - строковые шаблоны с подстановкой переменных для логов

      * **debug/** - шаблоны отладочных сообщений, разделённые по доменам

        * `channel.py` - отладочные логи для операций с каналами

        * `common.py` - общие отладочные логи, не привязанные к конкретному домену

        * `config.py` - отладочные логи для операций с конфигами

      * **info/** - шаблоны информационных сообщений, разделённые по доменам

        * `channel.py` - информационные логи для операций с каналами

        * `common.py` - общие информационные логи

        * `config.py` - информационные логи для операций с конфигами

      * `common.py` - общие шаблоны (например, описание прогресс-бара)

## Структура JSON файла каналов

Файл `channels/current.json` содержит информацию о Telegram каналах. Каждый ключ - **имя канала**, а значение - **объект с метаданными канала**.

### Пример JSON каналов

```json
{
    "channel_available": {
        "count": 5555,
        "current_id": 4444,
        "last_id": 4444,
        "state": 1
    },
    "channel_new_default": {
        "count": 0,
        "current_id": 1,
        "last_id": -1,
        "state": 0
    },
    "channel_will_be_deleted": {
        "count": 1234,
        "current_id": 5678,
        "last_id": -1,
        "state": -3
    },
    "channel_unavailable": {
        "count": 1000,
        "current_id": 3000,
        "last_id": -1,
        "state": -1
    }
}
```

### Подробное описание полей

* `count` - количество найденных V2Ray-конфигураций в канале.

  * `0` - значение по умолчанию, канал ещё не обработан или конфигурации не найдены.

  * `> 0` - количество найденных конфигураций в активном канале.

	> Значение `count` может расти до бесконечности по мере появления новых конфигураций в канале.

* `current_id` - ID текущего сообщения, с которого начинается сканирование.

  * `1` - значение по умолчанию, сканирование начинается с самого начала канала (старые сообщения).

  * `< 0` - взять последние `N` сообщений (новые сообщения).

    > Пример: если `last_id = 500` и `current_id = -100`, сканирование идёт с сообщения `≥ 400` до последнего `≤ 500`.

* `last_id` - ID последнего сообщения в канале.

  * `-1` - значение по умолчанию, канал временно/навсегда недоступен.

  * `> 0` - ID последнего доступного сообщения.

  	> Обновляется при каждом запуске сканирования.

* `state` - состояние канала (активный/неопределённый/неактивный).

  * `1` - активный канал, когда `last_id != -1`.

  * `0` - значение по умолчанию, состояние ещё не определено.

  * `-1` - неактивный канал, когда `last_id == -1`.

  > Если канал недоступен (`last_id == -1`), значение `state` с каждым запуском уменьшается на 1, показывая количество неудачных попыток доступа к каналу.

## Поддерживаемые протоколы

Файл с очищенными конфигурациями (`configs/v2ray-clean.txt`) содержит записи в одном из следующих форматов:

### **AnyTLS**

```text
anytls://password@host:port/path?params#name
anytls://password@host:port?params#name
```

### **Hy2 / Hysteria2**

```text
hy2://password@host:port/path?params#name
hy2://password@host:port?params#name
hysteria2://password@host:port/path?params#name
hysteria2://password@host:port?params#name
```

### **Shadowsocks / ShadowsocksR**

```text
ss://method:password@host:port#name
ss://method:password@host:port/path?params#name
ss://base64(method:password)@host:port#name
ss://base64(method:password)@host:port/path?params#name
ss://base64(method:password@host:port)#name
ssr://base64(host:port:protocol:method:obfs:base64(password)/?param=base64(value))
```

### **Trojan**

```text
trojan://password@host:port/path?params#name
trojan://password@host:port?params#name
```

### **TUIC**

```text
tuic://uuid:password@host:port/path?params#name
tuic://uuid:password@host:port?params#name
```

### **VLESS**

```text
vless://uuid@host:port/path?params#name
vless://uuid@host:port?params#name
```

### **VMess**

```text
vmess://base64(json)
vmess://uuid@host:port/path?params#name
vmess://uuid@host:port?params#name
```

### **WireGuard**

```text
wireguard://privatekey@host:port/path?params#name
wireguard://privatekey@host:port?params#name
```
